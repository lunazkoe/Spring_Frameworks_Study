### 싱글톤 패턴
- 클래스의 인스턴스가 딱 1개만 생성되는 것을 보장하는 디자인 패턴

### 싱글톤 패턴의 문제점
- 구현 코드 자체가 많아짐
- 의존관계 상 구체 클래스에 의존하게 됨 -> DIP / OCP 위반
- 테스트가 어려움
- private 생성자를 사용해야하므로 자식 클래스를 만들기 어려움
- 결국, 유연성이 떨어지는 안티 패턴으로 불리기도 한다..

### 싱글톤 컨테이너
- 스프링 컨테이너는 싱글톤 패턴을 적용하지 않아도, 객체 인스턴스를 싱글톤으로 관리
    - 즉, 객체를 1게 생성하면 더 생성하지 않는다!!

### 싱글톤 방식의 주의점
- **무상태로 설계**
- 특정 클라이언트에 의존적인 필드가 있으면 안됨
- 특정 클라이언트가 변경할 수 있는 필드가 있으면 안됨
- 가급적이면 읽기만 가능하면 좋음
- 필드 대신에 자바에서 공유되지 않는 지역변수, 파라미터, ThreadLocal을 사용해야함
- **공유 값을 설정하면 매우 큰 장애가 발생할 수 있다!!**

### @Configuration과 싱글톤
- 빈 등록 (생성자 주입)
    - 1. @Bean으로 된 메서드를 실행해서 반환되는 객체를 스프링 빈으로 등록
    - 2. 이때 반환되는 객체에 빈 객체가 의존성이 필요하다면
        - 빈 객체를 생성 후 등록
    - 3. 나중에 이미 등록해놓은 빈 객체의 생성 메서드를 부르려고 한다면
        - 있다면 만들지 않음
        - 없다면 만듦
    - 4. Autowired로 되어있는걸 먼저 만들려고 한다

### @Configuration과 바이트코드 조작의 마법
- @Configuration을 제거하면 @Bean 객체를 등록할 수는 있지만
- **싱글톤을 보장하지 않는다!!!**
- @Configuration을 붙인 설정 파일도 빈 객체로 등록되는데 이 때 보면 CGLIB라는 바이트 조작 라이브러를 사용해서 상속 받아 자식 클래스로 스프링 빈에 등록
    - 빈이 없다면 생성 / 있다면 생성되있던걸 반환하는 로직이 추가 되는 것으로 예상할 수 있음

- 만약 붙이지 않는다면 발생하는 이슈!!
    - AppConfig의 생성자 주입에서 호출되는 메서드는 그냥 순수 new해서 객체를 호출하는 것과 동일함
    - 따라서 스프링 빈으로 등록된 것이 주입되는게 아니라 완전히 다른 인스턴스임!!

- **그냥 스프링 설정 정보는 항상 @Configuration 사용하자!!**
